### Current-row reset while folder sizes update

There was a major bug:

* while folder sizes were calculating, repeated model rebuilds caused keyboard current row to reset to row 1/top
* Down arrow would move a few rows, then snap back

Several failed/partial approaches happened:

1. Tried preserving current row by path around `rebuildRowsFromController()`
   * failed due to current code structure

2. Tried preserving in `refreshDisplayedRows()`
   * first used `currentIndex`, but `currentIndex` was not in root scope
   * fixed to `fileListView.currentIndex`

3. Tried path-based `displayedRows`
   * failed because `displayedRows` was not in that scope

4. Tried index-only restore
   * improved behavior but still reset

Final successful fix:

* `refreshDisplayedRows()` now updates existing `ListModel` rows **in place** when the displayed row shape/order is unchanged.
* It checks:
  * same row count
  * same row path at each index
* If unchanged, it calls:

```qml
fileModel.setProperty(index, "name", row.name)
fileModel.setProperty(index, "kind", row.kind)
fileModel.setProperty(index, "sizeBytes", row.sizeBytes)
fileModel.setProperty(index, "sizeStatus", row.sizeStatus)
fileModel.setProperty(index, "modifiedSecs", row.modifiedSecs)
fileModel.setProperty(index, "path", row.path)
fileModel.setProperty(index, "isDir", row.isDir)
```

* Only if filter/sort/order/shape changes does it clear and rebuild the model.

All Shift variants should select the range between the selection anchor / current row and destination row.

Implementation concept:

### In `qml/FileListView.qml`

Adds signals:

```qml

signal boundaryMoveRequested(int direction, bool extendSelection)
```

Adds:

```qml
function pageStep() {
    return Math.max(1, Math.floor(fileList.height / Math.max(1, rowHeight)) - 1)
}
```

Adds handling in `Keys.onPressed` for:

```qml
Qt.Key_PageDown
Qt.Key_PageUp
Qt.Key_Home
Qt.Key_End
```

with `event.modifiers & Qt.ShiftModifier`.

### In `qml/main.qml`

Adds helpers:

```qml
function setCurrentIndexWithOptionalRange(newIndex, extendSelection)
function pageMoveCurrent(direction, extendSelection)
function boundaryMoveCurrent(direction, extendSelection)
```

Selection logic:

* If `extendSelection`:
  * uses `lastSelectedIndex >= 0 ? lastSelectedIndex : oldIndex` as anchor
  * selects all `rowPathAt(index)` between anchor and destination
  * keeps `lastSelectedIndex` as the anchor
* If not extending:
  * clears anchor with `lastSelectedIndex = -1`
* Always updates:
  * `fileListView.currentIndex`
  * `fileListView.containIndex(newIndex)`
  * `fileListView.forceListFocus()`

Adds FileListView instance handlers:

```qml
onPageMoveRequested: function(direction, extendSelection) {
    root.pageMoveCurrent(direction, extendSelection)
}

onBoundaryMoveRequested: function(direction, extendSelection) {
    root.boundaryMoveCurrent(direction, extendSelection)
}
```

## Important Current Snippets From User’s Source

### `qml/main.qml` around refresh before latest navigation patch

Known current successful model refresh after row-reset fix should resemble this concept:

```qml
function refreshDisplayedRows() {
    let rows = sortRows(filterRows(allRows))
    let sameShape = fileModel.count === rows.length

    if (sameShape) {
        for (let index = 0; index < rows.length; index += 1) {
            if (String(fileModel.get(index).path || "") !== String(rows[index].path || "")) {
                sameShape = false
                break
            }
        }
    }

    if (sameShape) {
        for (let index = 0; index < rows.length; index += 1) {
            let row = rows[index]
            fileModel.setProperty(index, "name", row.name)
            fileModel.setProperty(index, "kind", row.kind)
            fileModel.setProperty(index, "sizeBytes", row.sizeBytes)
            fileModel.setProperty(index, "sizeStatus", row.sizeStatus)
            fileModel.setProperty(index, "modifiedSecs", row.modifiedSecs)
            fileModel.setProperty(index, "path", row.path)
            fileModel.setProperty(index, "isDir", row.isDir)
        }
        return
    }

    let preservedIndex = fileListView.currentIndex
    fileModel.clear()
    for (let index = 0; index < rows.length; index += 1) {
        fileModel.append(rows[index])
    }

    if (fileModel.count > 0) {
        fileListView.currentIndex = Math.max(0, Math.min(preservedIndex, fileModel.count - 1))
    } else {
        fileListView.currentIndex = -1
    }
}
```

### `qml/FileListView.qml` current index design

The file list has:

```qml
property int currentIndex: fileList.currentIndex
readonly property int count: fileList.count
```

and:

```qml
onCurrentIndexChanged: {
    if (fileList.currentIndex !== currentIndex) {
        fileList.currentIndex = currentIndex
    }
}
```

Inside its internal `ListView`, there is likely:

```qml
onCurrentIndexChanged: fileListView.currentIndex = currentIndex
```

So external code should use:

```qml
fileListView.currentIndex
```

not plain `currentIndex`.

## Known Pitfalls From This Session

Do **not** use an unqualified `currentIndex` in `main.qml`; use:

```qml
fileListView.currentIndex
```

Do **not** refer to unqualified `displayedRows` from root-level refresh helpers unless confirmed in scope.

Avoid repeated:

```qml
fileModel.clear()
fileModel.append(...)
```

during folder-size updates, because that resets `ListView.currentIndex`.

Avoid repeated deferred row restore during size updates:

```qml
Qt.callLater(...)
```

if the restore is based on stale indices, because it fights keyboard navigation.
